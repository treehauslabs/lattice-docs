Ivy Networking

Peer-to-peer networking layer with an economic protocol. Kademlia DHT routing, TCP transport via swift-nio, fee-based content routing, credit line accounting, pin discovery, STUN NAT traversal, peer exchange (PEX), and XOR-distance-based storage with eviction protection.

.package(url: "https://github.com/treehauslabs/Ivy.git", from: "3.0.0")

Ivy Actor

actor

The main node. Manages connections, routes content requests, maintains the DHT, and tracks credit line balances with every peer via an economic layer.

Internally, Ivy maintains a blockCache (bounded dictionary of recently seen block data), a pinAnnouncements table (CID to list of known pinners), and an InventorySet (haveSet) tracking which CIDs the node has seen.

PeerConnection

A TCP connection to a single peer, built on swift-nio Channel.

public final class PeerConnection: @unchecked Sendable {
    public internal(set) var id: PeerID
    public let endpoint: PeerEndpoint
    public var messages: AsyncStream<Message>

    public static func dial(
        endpoint: PeerEndpoint,
        group: EventLoopGroup
    ) async throws -> PeerConnection

    public func send(_ message: Message) async throws
    public func fireAndForget(_ payload: Data)
    public func fireAndForgetMessage(_ message: Message)
    public func cancel()
}

Inbound messages are delivered via AsyncStream<Message>. The stream finishes when the connection closes.

NIO Pipeline

TCP bytes → MessageFrameDecoder → PeerChannelHandler → AsyncStream<Message> | | +-- length-prefix decode +-- feed to PeerConnection +-- Message.deserialize +-- close detection

MessageFrameDecoder handles buffering and 4-byte big-endian length prefix decoding, then deserializes the payload into a Message enum value.

Identify Protocol

On connect, each side sends an identify message containing its public key, the observed address of the remote peer, its own listen addresses, and a Curve25519 signature. The receiver verifies the signature against the claimed public key. If verification fails, the peer is disconnected. This also enables inbound connections (which start with a temporary ID) to be reassociated with the correct PeerID.

STUN NAT Traversal

actor

On startup, Ivy uses a STUNClient to discover the node's public IP address and port via STUN (Session Traversal Utilities for NAT). The discovered address is stored as publicAddress and included in identify messages to inform peers how to reach this node.

public struct ObservedAddress: Sendable, Equatable, Hashable {
    public let host: String
    public let port: UInt16
}

Default STUN servers: stun.l.google.com:19302, stun1.l.google.com:19302, stun.cloudflare.com:3478. Configurable via IvyConfig.stunServers.

Observed addresses are also collected from identify messages received from other peers. When two or more peers report the same external address, it is promoted to the node's publicAddress and the delegate is notified via ivy(_:didDiscoverPublicAddress:).

How K-Buckets Work

The 256-bit SHA-256 keyspace is divided into 256 buckets by Common Prefix Length (CPL) between the local node's hash and each peer's hash:

  • Bucket 0: Peers where the first bit differs (half the keyspace -- far away)
  • Bucket 1: Peers sharing 1 prefix bit (quarter of the keyspace)
  • Bucket 255: Peers sharing 255 prefix bits (extremely close neighbors)

Each bucket holds at most k entries (default 20). When full, the peer with the lowest Tally reputation is evicted.

Fee Model

Ivy uses a fee-bid model for content routing. The requester sets a total fee budget; each relay hop deducts its relayFee and forwards the remainder. The destination (the node that actually has the content) keeps whatever fee remains.

How Fee-Based Forwarding Works

  1. Request: Node A calls get(cid:fee:). This sends a dhtForward(cid, ttl, fee, target?, selector?) to the closest peers.
  2. Relay Check: Each relay node checks two gates -- behavioral (Tally reputation: shouldAllow) and economic (credit line: availableCapacity > 0).
  3. Local Hit: If the relay has the block in its cache or local storage, it responds immediately and earns the full remaining fee.
  4. Forward: If the relay does not have the block and fee > relayFee, it deducts relayFee, records a pending forward, and sends the request onward to the closest peer toward the target hash.
  5. Fee Exhausted: If fee <= relayFee and the relay cannot serve the content, it sends feeExhausted(consumed:) back.

Pay-on-Success

Fees are only charged when a response successfully relays back through the chain. When a .block response arrives at a relay node:

  1. The relay forwards the block to the upstream requester.
  2. The relay calls ledger.earnFromRelay(peer:amount:) to credit its relay fee against the upstream peer's credit line.
  3. The block is cached locally for future requests.

If no response arrives (timeout), no fees are charged.

Targeted Requests

dhtForward supports an optional target (a peer hash to route toward) and selector (a path selector for sub-DAG traversal). When target is set, the request is routed toward that peer's hash neighborhood rather than the CID's hash neighborhood. This is used by get(cid:target:fee:) after discovering a pinner.

A ----dhtForward(cid, fee=20)----> B ----dhtForward(cid, fee=19)----> C ----dhtForward(cid, fee=18)----> Z (has block) | | | | <------block(cid, data)-------- | <-------block(cid, data)-------- | | B earns 1 (relayFee) | C earns 1 (relayFee) Z earns 18 A receives block

Credit Lines

Every peer pair shares a CreditLine managed by the CreditLineLedger actor. Credit lines track the net balance of services rendered between two peers and determine when settlement is required.

Trust Bootstrapping

Initial credit line thresholds are derived from the peer's public key difficulty -- the number of trailing zero bits in the key's hex representation. Keys with more trailing zeros required more computational work to generate, so they receive higher initial trust:

threshold = baseTrust(publicKey) * baseThresholdMultiplier

Subsequent successful settlements grow the threshold logarithmically. Missed settlements halve it.

Balance Reconciliation

Peers can exchange balanceCheck(sequence:balance:) messages to verify they agree on the credit line state. Divergences are logged for investigation. The balanceLog message allows a peer to request a replay of recent operations for auditing.

Pin Discovery

Pin discovery allows nodes to advertise that they store (pin) certain content, and other nodes to find those pinners.

Publishing a Pin

  1. Call save(cid:data:pin:true) or publishPinAnnounce(rootCID:selector:expiry:signature:fee:).
  2. This sends pinAnnounce messages to the k closest peers to the CID's hash. Those peers store the announcement in their pinAnnouncements table.
  3. The receiving peer replies with pinStored(rootCID:) to confirm storage.

Finding Pinners

  1. Call discoverPinners(cid:fee:).
  2. This sends fee-based findPins(cid:fee:) messages toward the CID's hash neighborhood.
  3. Relay nodes that have stored pin announcements respond with pins(announcements:) and earn the fee. Otherwise they forward (deducting their relay fee) toward closer peers.
  4. Results arrive via the delegate callback ivy(_:didReceiveMessage:from:) as .pins messages.

Fetching from a Specific Pinner

Once you know a pinner's public key (from discoverPinners), use get(cid:target:fee:) to route the request toward that specific peer rather than the CID hash:

let pinners = await ivy.discoverPinners(cid: myCID)
if let first = pinners.first {
    let target = PeerID(publicKey: first.publicKey)
    let data = await ivy.get(cid: myCID, target: target)
}

Settlement

When a credit line's |balance| reaches its threshold, the debtor must settle. Ivy supports two settlement mechanisms:

Mining Settlement

  1. The creditor issues a miningChallenge(hashPrefix:blockTargetDifficulty:noncePrefix:) to the debtor.
  2. The debtor performs proof-of-work and responds with miningChallengeSolution(nonce:hash:blockNonce:).
  3. The creditor verifies the work. The work value is 2^(trailingZeroBits(hash) - 16) ivy. This amount is credited against the outstanding balance via recordPartialSettlement.
  4. If the balance reaches zero, the credit line is fully settled via recordSettlement(), which resets the balance and grows the threshold.

On-Chain Settlement

  1. The debtor submits an on-chain transaction and sends settlementProof(txHash:amount:chainId:) to the creditor.
  2. The creditor records the partial settlement and checks if the balance is now settled.

Missed Settlements

If a debtor fails to settle, the creditor calls recordMissedSettlement(peer:), which halves the credit line threshold. This limits the debtor's future capacity until trust is rebuilt.

Peer Exchange (PEX)

PEX provides network resiliency by allowing peers to discover new peers through their existing connections, reducing dependence on bootstrap nodes.

  1. Periodically (default every 120s), a node selects a random connected peer and sends pexRequest(nonce:).
  2. The recipient responds with pexResponse(nonce:peers:) containing up to pexMaxPeers (default 16) peer endpoints, selected from peers closest to the requester's hash (excluding loopback and local addresses).
  3. The requester adds new peers to its routing table and attempts to connect.

Zone Sync and Replication

Two background processes ensure data redundancy across the DHT:

Zone Sync

Periodically (default every 30 minutes), the node asks its closest DHT neighbors for their zone inventory via getZoneInventory(nodeHash:limit:). Any CIDs the node does not have are fetched.

Proactive Replication

Periodically (default every 5 minutes), the node samples stored CIDs and probes zone-relevant peers via haveCIDs to check replica counts. Under-replicated blocks (fewer than replicationMinCopies, default 3) are pushed to the closest peers that lack them.

Local Discovery

On Apple platforms (#if canImport(Network)), Ivy can discover peers on the local network via Bonjour/mDNS:

  • Advertising: Announces the node's service type, port, and public key as a TXT record
  • Browsing: Discovers other Ivy nodes on the LAN and auto-connects
  • Service type: Configurable via IvyConfig.serviceType (default "_ivy._tcp")
Linux
Local discovery requires Network.framework and is only available on Apple platforms. On Linux, use bootstrapPeers in the config to specify known peers.

Content Routing Flow

When get(cid:fee:) is called:

  1. Local check: Block cache and local CAS storage are checked first.
  2. Fee-based DHT forward: dhtForward(cid, ttl, fee) is sent to the closest peers in the CID's hash neighborhood.
  3. Dual-gated relay: Each relay checks Tally reputation (behavioral) and credit line capacity (economic) before forwarding.
  4. Relay fee deduction: Each hop deducts relayFee (default 1) from the fee budget and forwards the remainder.
  5. First response wins: The first .block response resolves the request; the timeout (default 15s) resolves to nil.
  6. Pay-on-success: Fees are only charged when the response successfully relays back through the chain.
  7. Caching: Responded blocks are cached locally in the block cache and optionally stored in the local CAS worker.
