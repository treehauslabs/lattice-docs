DiskBroker Storage

SQLite-backed Volume broker with WAL journaling, Volume-granular storage, ref-counted pin tracking, and automatic eviction of unpinned data.

.package(url: "https://github.com/treehauslabs/DiskBroker.git", from: "1.0.0")

DiskBroker

actor VolumeBroker

Persists Volumes to a SQLite database. All data, entry associations, pin state, and metadata are stored in four tables within a single database file.

public actor DiskBroker: VolumeBroker {
    public init(
        directory: URL,
        walMode: Bool = true
    ) throws
}

Parameters

directory URL Directory for the SQLite database file. Created automatically if it does not exist.
walMode Bool Enable WAL (Write-Ahead Logging) journal mode. Default true. Provides concurrent reads during writes.

SQLite Schema

DiskBroker uses four tables to store Volume data, entry associations, pin state, and metadata.

cas_data

Raw content-addressed data storage. Each entry maps a CID to its serialized bytes.

CREATE TABLE cas_data (
    cid   TEXT PRIMARY KEY,
    data  BLOB NOT NULL
);

volume_entries

Maps Volume roots to their constituent entry CIDs. A Volume's entries are the set of rows matching its root.

CREATE TABLE volume_entries (
    root  TEXT NOT NULL,
    cid   TEXT NOT NULL,
    PRIMARY KEY (root, cid),
    FOREIGN KEY (cid) REFERENCES cas_data(cid)
);

volume_pins

Ref-counted pin tracking per root and owner, with optional expiry.

CREATE TABLE volume_pins (
    root        TEXT NOT NULL,
    owner       TEXT NOT NULL,
    count       INTEGER NOT NULL DEFAULT 1,
    expires_at  REAL,  -- Unix timestamp, NULL for no expiry
    PRIMARY KEY (root, owner)
);

volume_metadata

Tracks when each Volume was stored, for diagnostics and ordered enumeration.

CREATE TABLE volume_metadata (
    root       TEXT PRIMARY KEY,
    stored_at  REAL NOT NULL  -- Unix timestamp
);

Methods

MethodBehavior
hasVolume(root:) Check if a row exists in volume_metadata for the given root.
fetchVolumeLocal(root:) async Join volume_entries with cas_data to reconstruct the full Volume (root + all entry data).
storeVolumeLocal(_:) async Insert all entries into cas_data, record associations in volume_entries, and write a row to volume_metadata. Wrapped in a single transaction.
pin(root:owner:count:ttl:) Upsert into volume_pins. Increments existing count if a pin for this root+owner already exists.
unpin(root:owner:count:) Decrement pin count. Deletes the row if count reaches zero.
unpinAll(owner:) Delete all rows in volume_pins where owner matches.
evictUnpinned() Delete all Volumes (from all four tables) that have no active, non-expired pins. See below.
checkpoint() Force a WAL checkpoint, merging the write-ahead log back into the main database file. Reduces WAL file size.

evictUnpinned()

Identifies all Volume roots that have no active pins (either no rows in volume_pins, or all rows have count = 0 or expires_at < now). For each such root:

  1. Delete rows from volume_pins for the root
  2. Collect entry CIDs from volume_entries
  3. Delete volume_entries rows for the root
  4. Delete volume_metadata row for the root
  5. Delete cas_data rows for entry CIDs not referenced by any other Volume

Step 5 ensures that shared entries (CIDs referenced by multiple Volumes) are only deleted when no Volume references them.

Shared entry protection
Because multiple Volumes can share entry CIDs (e.g., unchanged Merkle subtrees), evictUnpinned() performs a reference check before deleting from cas_data. An entry is only deleted when zero remaining volume_entries rows reference it.

WAL Journaling

DiskBroker defaults to SQLite WAL (Write-Ahead Logging) mode, which provides:

  • Concurrent reads: Readers do not block writers and vice versa. Multiple actors can fetch Volumes while another stores.
  • Crash safety: Incomplete transactions are rolled back on next open. No partial Volumes are ever visible.
  • Reduced fsync: Writes append to the WAL file sequentially, which is faster than random-access updates to the main database.

checkpoint()

The WAL file grows unboundedly until a checkpoint merges it back into the main database. Call checkpoint() periodically (e.g., after eviction) to reclaim WAL space. SQLite also auto-checkpoints when the WAL reaches ~1000 pages.

// After bulk eviction, merge WAL back to main DB
await diskBroker.evictUnpinned()
await diskBroker.checkpoint()
Disable WAL for read-only
If the database is opened read-only (e.g., for inspection tools), pass walMode: false to avoid WAL file creation.
